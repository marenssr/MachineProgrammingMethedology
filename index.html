Controls Engineering Standard · Rev 1.0
Machine Programming Methodology

This is a programming standard for controls architecture built on ISA-88 and PackML principles, covering the full stack from system design through HMI deployment: architecture and design hierarchy, state machine design (Machine Manager and Module Manager), device management via Control Modules, alarm classification, and operator interface standards.

It defines a consistent methodology for Beckhoff TwinCAT3 and Ignition SCADA deployments, with patterns that are adaptable across any IEC 61131-3 platform.

Architecture · 1.0
System Overview

This standard implements a multi level architecture across a total automation system. Each level has a defined purpose and is not necessarily dependent on the others. The core focus is automated machines and their programming architecture.

System Levels
SystemDescription
Operating System (Software)Overall brains of the processes. Key business decisions, process setpoints, traceability, and condition monitoring. Strings together operations via sequence requests but does not directly control machines, machine control is left to the PLC.
HMI: Ignition SCADAResponsible for direct control of machines and circumvention of upper level software when needed. Used for operator interfaces during commissioning and maintenance, visual SCADA overviews, and variable historization to a SQL database.
Machine (PLC)A combination of electrical, mechanical, and programming components that allow processes to run. Machines can be Discrete, Continuous, or Batch. This guide defines how to design the programming component.

Design Hierarchy

The design hierarchy is drawn from two models: a Physical Model (physical processing and control equipment grouped by purpose) and a Procedural Model (operations that execute controlled actions in ordered sequences).

Physical Model: Equipment Hierarchy
Machine Manager (FB_Machine)
Module Manager (FB_Module)
Control Module
Control Module
Control Module
Equipment Class — drives child actions (solid border)
Device Class — pre-programmed, machine-agnostic (dashed)
Key Rules
Parent (master) → Child (slave) CommandsA "parent" is always a Machine or Module instance. A "child" can be either a Module or Device. Commands routed through the parent drive procedural actions of its children.
Equipment ClassEquipment can operate as a Parent and drive actions through sequential or conditional logic. Machine specific, but extends the library FB_Machine or FB_Module base class.
Device ClassDevices are child components with pre programmed actions common across machines. On abort or stop, devices are commanded to a safe state by their parent.
Architecture · 1.2
Standard Definitions

Nomenclature is foundational to consistent software design. All terms below are applied uniformly across PLC code, HMI labels, and documentation.

TermDescription
MachineAn entire production machine based on a 10 state model including pausing and homing. Every machine extends FB_Machine. Use when a complex homing sequence is required to return to a known good state.
ModuleA simpler production machine based on a 4 state model. Extends FB_Module. Distinguishable because the Home option is not available.
SequenceAn ordered list of executable device actions. Only used in automatic mode of machines or modules.
OperationA combination of multiple sequences achieving an overarching objective. Typically defined by upperl evel SCADA, or on the machine itself.
Control Module (CM)A device or combination of devices operated as a single action. The lowest level that carries out basic functions (valves, clamps, motors). Core code lives here and is transferable machine to machine without changes.
FunctionInterfaces between sequences and control modules. Calls specific device actions and waits for a response before advancing. Only one function per machine may be active at a time.
Conditional CommandAny programmed device actuation done outside of a sequence, based on specific conditions. Used in auto mode only. Does not depend on machine history.
Machine StateA condition of the machine determined by previous inputs, which defines actions on following inputs. States determine whether the machine is in Auto or Manual mode.
Machine Mode (Auto)Executes sequences issued by SCADA, the HMI, or conditional logic based on sensor status.
Machine Mode (Manual)Allows the operator to manipulate devices independently through the HMI. Modes are determined by state and passed down to child components.
Operator Command (ocmd)Commands accepted by components in Manual mode. Issued via the HMI operator interface.
Program Command (pcmd)Commands accepted by components in Auto mode. Issued programmatically by sequences or conditional logic.
Control Command (ccmd)Accessible in all modes. Includes Reset, Enable/Disable Simulation, Enable/Disable Bypass, Enable/Disable Manual Override.
Architecture · 1.3
Sequencer & Functions

The sequencer enables sequential control of a defined group of control devices via ordered steps. It allows creation of steps executable from multiple code sections without duplicate logic. Only one sequence may run at a given time.

Sequencer Rules
Permissives requiredEach sequence must define permissives and all must be true before the sequence may start. Sequences do not have interlocks.
Auto execute in transient statesCertain sequences execute automatically in transient states if permissives are met. Execute only in Acting States by request otherwise.
Auto mode onlySequences execute only in states that correspond to automatic mode which are all states except Paused, Stopped, and Aborted.
Sequencer Function Responsibilities
Command
Issue device actions
Commands a control module, multiple modules, or executes branches. Can issue parallel commands, custom logic is performed external to the function.
Monitor
Track failures & timeouts
Monitors failures or timeouts of the active function and surfaces fault codes up the hierarchy.
Advance
Set complete signal
Sets a complete signal to advance to the next sequence index, or branch to a different location within the sequencer array.
Retry
Retry logic
Attempts to fix the problem via external logic, or as simply as reissuing the previous control module commands.

Machine Manager

The Machine Manager transitions the machine between states (and modes). Based on a 10-state model derived from PackML, reduced from 17 states. Click any state node to view its description and valid commands.

MACHINE MANAGER STATE MACHINE (click a state to explore)
Acting State — executes code and sequences
Wait State — stable, awaits command or interrupt
Entry / Exit Permissives

Each Acting State has entry and exit permissives. A state can only be entered if entry permissives are met, it exits by command or when State Complete permissives are met. Naming convention: PermHome, PermHomeComplete.

Interrupts such as E-STOP or safety breach cause state transitions before a complete signal is received, for example an E-STOP while Running transitions immediately to Aborting.

States · 2.1
Module Manager

The Module Manager functions like the Machine Manager in simplified form — a 4 state model. Modules are distinguishable because the "Home" option is not available. Click any state to explore.

MODULE MANAGER STATE MACHINE (click a state to explore)
Key Differences from Machine Manager
No HomingModules have a Starting state that transitions directly into Started. No complex homing sequence, may only enforce valve positions or known device states.
Stopping vs. AbortingOn E-STOP while in Started state, the module transitions to Stopping (not Aborting). Stopped does not guarantee a safe state.
RecoveryThe only automated path to reset a known state is to Start the module again. A Reset command can also be issued by the operator.
States · 2.2
Substates & Modes

Substates are custom defined states inside Running or Starting that provide more granularity than the standard state machine. Each substate may or may not run a sequence within it.

Substate Types
Transient Substates
Typically run sequences
Execute sequences within them and move to another state upon completion. Do not linger awaiting a command.
Final Substates
Conditional logic only
Do not run sequences. Execute conditional logic that responds to real-time machine and sensor conditions.
Conditional Commands

A conditional command controls devices or modules asynchronously and is not dependent on any history of a machine or module — it does not live inside a sequence. It can and likely does depend on current states and substates.

Auto vs. Manual Mode
Auto ModeExecutes sequences issued by SCADA, the HMI, or conditional logic. Active in Running, Homing, and other acting states.
Manual ModeAllows the operator to manipulate devices independently through the HMI. Active in Paused, Stopped, and Aborted states.
Manual OverrideA child component can temporarily overwrite the parent mode. Override is available in both modes but is only useful in Auto. Must be time limited at a configurable access level.
Devices · 3.0
Control Modules

Control Modules are encapsulated code (like an AOI in Ladder Logic or a Function Block in Structured Text) that directly controls a single device. Designed to be validated independently and transferred machine to machine without core code changes.

Required CM Properties
PermissivesAllow a command to initiate an action. Default to TRUE if not applicable.
InterlocksStop an action currently being executed. Default to TRUE if not applicable.
Command InputsSeparate inputs for Program Commands (pcmd), Operator Commands (ocmd), and Control Commands (ccmd).
Command UnlatchingCommands must be unlatched during the processing of actions to prevent repeat triggering.
Fault FlagAn error sets the module's 'faulted' flag, which is a logical OR of all active errors. Built in monitoring required.
Statistics / MetricsDesign in when applicable: uptime vs. downtime, actuator transit time, cycle count.
Status BitsReadable by outside programs for monitoring and diagnostics.
Reset Command (ccmd)Clears internal tags, status bits, and input, but not configuration.
Devices · 3.1
Permissives & Interlocks

Permissives and interlocks are the primary mechanisms to prevent machine damage and visualize why a machine is not proceeding. Both must be exposed on the HMI with text descriptions per bit.

Permissives
Prevent something from starting
Prevents a command from initiating its associated action, prevents a transition state from completing, or prevents a Sequence Step from advancing. Evaluated before an action begins.
Interlocks
Stop an action in progress
A subset of permissives. Primary purpose is to stop an action while it is operational. Also evaluated at command time to prevent initiating actions known to fail.
Interlock Scenarios
ScenarioUse Case
1Machine Manager in Auto, auto interlock drops (E-STOP, loss of air) → machine goes to manual. Immediately stops the active sequence.
2Prevent actuators from crashing during movement, position-based interlocks block motion if an adjacent device is in the path.
3Coordinated servo actions, stop movement if servos become decoupled or lose synchronization.

Errors & Recovery

Error is a superset that includes device faults and other conditions such as receiving an operator command in auto mode. Each error requires a defined recovery strategy.

Fault Types
EventCan CauseExpected ResultExpected Resolution
Device FaultSequencer Function Fault, Manager FaultDevice turns Red on HMI. Diagnostic event logged.Reset Device or Reset its Parent Manager.
Sequencer Function FaultSequencer FaultGenerates a Sequencer Fault with fault code. Logged at sequence and manager level.If salvageable: pause, address, then Proceed or Retry. Otherwise: Abort.
Sequencer FaultManager FaultFault code displayed to user and logged. Propagated up to Manager.Operator Reset, or reset by next successful sequence call.
Machine/Module Manager FaultTop level fault, may require software handlingFault code displayed and logged for the equipment.(1) Error condition disappears, (2) Software clears the error, or (3) Homing/Starting state.
Recovery Strategy Framework
Auto Recoverable (Machine)
Fully automated
Machine detects and resolves the condition without operator intervention. Preferred when sensorization allows for it.
Auto Recoverable (Software)
Software initiated
Higher level software service detects the fault and issues the appropriate recovery command to the machine.
Manual Recovery
Operator intervention
Operator confirms machine state before recovery proceeds. Escalated via SCADA alarms to the correct individual.
Fault Propagation Rule
Inherit only if halting product
Manager inherits child fault only if the fault causes it to halt product processing. Otherwise, lower level fault is displayed on HMI but not passed up.
Alarms & Safety · 4.0
Alarm Hierarchy

Alarms are classified by priority. General hierarchy: Critical Alarm > Machine Fault > Warning > Diagnostic. A machine cannot be faulted and running at the same time — use warnings or diagnostics for non-halting alerts.

Critical Alarm · Highest Priority
Safety device or critical function failure
Any critical safety device (E-STOP, safety mat, gate, light curtain) that causes machine stoppage, or critical functions such as Air and Power.
Machine Fault · High Priority
Event that has stopped the machine
Generally a sequence fault or state fault. Must show as RED on the UI. One alarm message per machine fault. A machine cannot be faulted and running simultaneously.
Warning · Medium Priority
Condition that may stop the machine
Any alarm that may cause a future stoppage (low limit switches on tanks, temperature warnings). Shows as YELLOW on the UI.
Diagnostic · Informational
Possible issue, not a confirmed stop cause
Indicates an issue but cannot be determined as the reason the machine stopped. Does not halt the machine.
Process Alarms — Continuous Machines
LevelTypical Response
HiHiInterlock out the device — e.g. if Supply Tank reaches HiHi, interlock out the supply pump to that tank.
HiDrive a machine event — e.g. command a stop sequence to return devices (pump, valves).
LoDrive a machine event — e.g. command the nutrient inject to start a fill process.
LoLoInterlock out a device — e.g. if supply tank reaches LoLo, interlock out the supply pump.

Safety Behaviors

Expected machine behaviors for each safety event type, with recovery procedures. All machines should be recoverable from any safety event without manual repositioning wherever possible.

ActionExpected ResponseRecovery
Machine E-STOPMachine commands an abort. All devices go to safe state.Reset safety device. Home the machine or Start the module.
Momentary Request to EnterMachine completes its current process or immediately pauses all devices. Programmer must think through all scenarios.Satisfy all safeties and press Proceed. Machine continues from previous state and operation.
Request to Enter (Override)Immediate Pause. Operates as expected response for protected or monitored space.Satisfy all safeties and press Proceed.
Protected Space DetectionAny energized device in the protected space is de-energized (may be entire machine). Devices commanded to safe states.Satisfy all safeties and press Proceed.
Monitored Space DetectionEnergized devices in monitored space must be de-energized. Devices outside must not enter without safe positioning or a protected space safety detection to continue.If devices in monitored space: satisfy safeties, Proceed. If not: devices can resume at full speed after safety reset.
Zone Migration (Monitored → Protected)Energized device in protected space is de-energized. Machine commands devices to safe states.Satisfy all safeties and press Proceed.

HMI · 5.1
Layout & Required Features

A common machine UI design reduces training time across operators, maintenance technicians, and engineers. The following structural elements are required for all production equipment deployments.

Structural Elements
ElementSpecification
Primary HeaderGeneral navigation and user information. Docked at the bottom of the screen.
Secondary HeaderZone specific control: Logo, Machine Name, State, Safety Status, Mode, Interlocks/Permissives, and control buttons (Home, Start, Abort, Reset). Docked at the top.
Navigation BarDocked on the left hand side.
Device Control PanelDocked on the right side. Opens as a sidebar when clicking a device on the Machine View.
Required Tabs
Machine View / P&ID Tab
Discrete: Machine View · Continuous: P&ID
All sensors and devices clearly visible. Faulted devices turn Red. Selecting a device opens the control sidebar, indicated by a blue border. Sequencer: current sequence step must be visible; sensors highlighted if a step is hanging waiting for a specific value.
Commission Tab · Password Protected
Commissioning & maintenance features
All features needed for commissioning or maintenance but not continuous operations. Editing offsets, calibrating, controlling via sequences not triggered by upper level software.
Safety Tab
Safety zone overview + Request to Enter
View of all safety devices in a safety zone. Contains Request to Enter functionality that brings equipment to a controlled stop in an auto recoverable state.
Alarms Tab
Active Alarms + Alarm History
Alarm summary sorted by priority. Alarm history tied to a database. Two sub tabs: Active Alarms and Alarm History.
Device Faceplate Rules
Auto / Manual ToggleAll device interfaces must allow toggling between Manual and Auto while the parent is in Auto. Manual override on a child is time limited and configurable by access level.
Commands AvailableOnly Operator Commands (ocmd) available from the UI. Program Commands (pcmd) are not exposed to operators.
TrendsInclude historical trends wherever room allows. Prioritize process equipment and sensors for operational context.

Naming Conventions

Naming conventions drive consistency across PLC code, HMI labels, permissive/interlock descriptions, and physical device labels. Physical tags in the field should match these conventions so commissioning and maintenance teams can locate devices without ambiguity.

Human Readable Description Convention
Format[TYPE_ABBREV]_[DeviceDescription]_[ActionOrSensorType]
Digital Input ExampleDI_PusherReturn_PRX — proximity sensor on a pusher actuator return position
VFD ExampleVFD_InfeedConveyor — variable frequency drive for an infeed conveyor
HMI Labels OnlyUse only the device description — not the full tag. A tower stop sensor displays as Stop Sensor, not DI_StopSensor_PRX.
Equipment Tagging

Equipment tags use the full human-readable name of the equipment. These are capital-purchased systems with multiple devices, instruments, and components operating together.

Motor, Valve & Instrumentation Tagging
P&ID ExistsDevice tagging on HMI overview and IO screens follows the existing P&ID convention exactly.
No P&IDHMI overview uses human-readable formats. IO screen uses human readable and/or schematic reference descriptions.
